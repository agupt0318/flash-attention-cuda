flash-attention-cuda

A from-scratch CUDA implementation of FlashAttention (Dao et al., 2022). Source: github.com/agupt0318/flash-attention-cuda

1. The problem

Attention takes n word vectors and computes a score between every pair: an n×n matrix. The straightforward implementation stores that matrix in GPU main memory (HBM), applies softmax to it, and multiplies by V. The arithmetic is not the bottleneck. Moving the matrix to and from main memory is: memory traffic and memory footprint both grow with n2, while everything else in the model grows roughly with n.

Each doubling of sequence length quadruples the memory. A model with 32 heads (as in the benchmark below) needs 32× these amounts.

2. What we are trying to solve

Compute attention with the same numerical result as the straightforward version, without ever storing the n×n matrix in main memory. Extra memory use should stay constant in n (a fixed set of running values), so long sequences stop being limited by the score matrix.

3. How it works

Softmax normally needs all n scores of a row before it can produce any output. The paper's observation is that it can be computed incrementally. For one query row, keep three running values:

  • m: the largest score seen so far
  • ℓ: the sum of exp(score − m) over the scores seen so far
  • acc: the output accumulated so far (unnormalized)

K and V are processed in tiles small enough to fit in on-chip memory (SRAM, ~100× faster than HBM). Each tile updates the three values; if a new tile contains a larger maximum, the history is rescaled by exp(m_old − m_new) before the tile is folded in. After the last tile, dividing acc by ℓ gives exactly the softmax-weighted output. Scores are consumed as they are produced and never written to main memory. This is Algorithm 1 in the paper; the kernel in this repo keeps m, ℓ, and acc in registers.

Read plainly:

  • At N=512 the naive version is the fastest of the four. The method only pays off once the score matrix is large.
  • torch SDPA is about 5% faster than this repo's CUDA kernel at every length. That is the expected result; SDPA is the production implementation of the same algorithm.
  • The memory difference is the point: 4.13 GiB vs 0.13 GiB at N=4,096. At N=8,192 with this batch size the naive version would need more than the T4's 16 GiB and fail; the streamed versions are unaffected.
  • Correctness: CPU tests pass at 1e-6 against a float64 reference, GPU tests at 5e-5, and the backward pass passes torch.autograd.gradcheck.

6. Why this exists

This is an educational implementation. The goal was to understand IO-aware kernel design by building it: the CPU version proves the algorithm against a float64 reference, the CUDA kernel implements it with explicit tiles and registers, the Triton version shows the same algorithm through a compiler, and the benchmark measures where the method wins and where it does not.

It was also written without local GPU hardware: the algorithm was validated on CPU, the kernel compiled in CI, and the first actual execution was on a free Colab T4 (all tests passed on that run). A profiler pass on the T4 then led to v2 of the kernel, which splits each row across two threads to halve per-thread register use. For real workloads, use torch.nn.functional.scaled_dot_product_attention; this repo is for reading.
